iOS backdoor

by William Harrington

Feb 17, 2016

Introduction

The FBI is taking unprecedented legal action that will force Apple to create a "backdoor" for iOS. That term may be a little of a misnomer because really what the FBI wants is the ability to crack an iOS device without having to worry about other security features.

A recent CNN news article discusses what exactly is transpiring between Apple and the FBI. In this article, they point out one of the most basic security features of iOS that is stopping the FBI from obtaining information from the mobile devices of the shooters from the mass shooting that took place in San Bernandino, CA December 2015.

"Apple's operating systems included an auto-erase function that, when enabled, would result in the information on the phone being permanently wiped after 10 failed attempts at inputting the passcode, the government wrote in documents seeking the order."

First off, it is pathetic that this is stopping the FBI. If you're a computer or electrical engineer, you are probably already cracking up. Second, modifying this feature will compromise all iOS users and I'd like to demonstrate why. I will do this in a way that even people who are not technically adept should be able to understand.

Disclaimer

The demonstration that follows is a conjecture. I do not proclaim to be an iOS expert or security expert. I am a computer engineer. I have developed apps for iOS professionally and used iOS devices extensively. That is the knowledge base I am making these conjectures from.

Demonstration

Passcodes

From my experience, the passcode on any iOS device consists of 4 integers that range from 0 to 9. If you know some basic mathematics, its easy to see that this means there are \(10^{4}\) or 10,000 combinations. The FBI would like to be able to try every possible combination until the actual passcode is found without having to worry about erasing the data on it. This is called a brute force attack.

Electronic entry

In its current state and as far as I know, iOS doesn't allow for these entries to happen electronically, which means someone needs to physically enter the passcode on the device. It should be obvious that this would be tedious and prone to error if you are just going to go through entering every possible passcode. The FBI would like iOS to be modified to accept electronic entries for the passcode.

What if the FBI gets what they want?

What the FBI is proposing is a modification of iOS that can be injected in an iOS compatible device that would allow it to accept the passcode electronically and bypass the 10 attempt limit. So just how easy would it be to hack an iOS device if I was able to do what they're proposing?

Simulating your device

Now let's define a simplified device and use it to see what it might be like to crack the passcode using brute force.

Please note this is not a 'clock accurate' simulation and that the delays added for I/O that are present in any electronic device are based on the computer running the code. In other words, this is not the proper way to model it, but for the sake of argument its close enough.

# an overly simple definition of an iOS device
class myDevice:
    def __init__(self, passcode='0000'):
        """
            This function initializes the device with the following input parameters
            
            :param input passcode: passcode for device, by default 0000
        """
        
        # the 4 integer passcode
        self.passcode = passcode
        
        # the data on the device, always starts with none
        self.data = None
        
        # this indicates whether the device is open
        # it is simply a control mechanism that
        # all functions will use to determine
        # their behavior
        self.open = False
        
    def store_data(self, data):
        """
            This function stores data on the device
            
            :param input data: data to store on device
        """
        
        # if self.open is True, set data
        if self.open:
            
            # 10ms delay
            time.sleep(.01)
            
            print '[+] Data stored'
            self.data = data
            
        else:
            print '[!] Access denied'
            
    def read_data(self):
        """
            This function reads data on the device
        """
        
        # if self.open is True, return data
        if self.open:
            
            # 10ms delay
            time.sleep(.01)
            return self.data
        
        else:
            print '[!] Access denied'
            
    def open_device(self, passcode):
        """
            This function opens device when the correct
            passcode is supplied
            
            :param input passcode:
        """
        
        # if passcode provided is equal to self.passcode
        # set self.open to True
        if passcode == self.passcode:
            print '[-] Device unlocked'
            self.open = True
            
    def close_device(self):
        """
            This function closes the device
        """
        print '[+] Device closed'
        self.open = False

Hacking your device

So far we've defined a device, instantiated it, created a completely random passcode for our device, opened it up and put some data on it, read the data and then closed the device.

Now let's envision this hypothetical situation: You are not in possession of your device because you lost it, I am now in possession of your device and I am not somebody that you want to be in possession of your device, and I can send passcodes to it electronically without having to worry about any other security features other than the passcode itself.

How long is it going to take me to get my hands on the data in your device?

Do you remember all those passcodes that I generated earlier in 13ms? I'm going to go through those one at a time and try to open the device. I will be sending passcodes electronically which means I'll be using Serial communication. Now the latency of such communication varies so I will just assume that for each character I send there is a 10ms delay.
